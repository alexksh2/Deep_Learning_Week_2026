 Auth storage

 Register writes new user.

 - Redirect unauthenticated users to /login

  The key pieces in lib/alpaca.ts:

  // getPositions() just calls alpacaFetch with the right path
  export async function getPositions() {
    return alpacaFetch<AlpacaPosition[]>("/v2/positions")
  }

  // alpacaFetch attaches the API keys (from .env.local) and hits Alpaca
  export async function alpacaFetch<T = unknown>(path: string, init?: RequestInit): Promise<T> {
    const base = path.startsWith("/v2/stocks") ? DATA_URL : BASE_URL   // paper-api.alpaca.markets
    const res = await fetch(`${base}${path}`, {
      ...init,
      headers: { ...authHeaders(), ... },   // injects ALPACA_API_KEY / ALPACA_SECRET_KEY
      cache: "no-store",
    })
    const body = await res.text()
    return JSON.parse(body) as T            // ← this is what becomes `positions`
  }

      "current_price": "523.10",
      "unrealized_pl": "16.00",
      ...
    }
  ]

⏺ Breaking it down piece by piece:
                                                                                                   
  The function signature            
  export async function alpacaFetch<T = unknown>(
    path: string,                                                                                   
    init?: RequestInit,
  ): Promise<T>                                                                                     
  - <T = unknown> — generic type, caller says what shape the response will be. getPositions() passes
   AlpacaPosition[], so T becomes that
  - init? — optional, same options object that native fetch accepts (method, body, etc.). The ?
  means you can omit it
  - Promise<T> — it's async so it always returns a Promise, which resolves to whatever T is

  ---
  Picking the base URL
  const base = path.startsWith("/v2/stocks") ? DATA_URL : BASE_URL
  Ternary — if the path begins with /v2/stocks, use data.alpaca.markets, otherwise use
  paper-api.alpaca.markets. Just a string assigned to base.

  ---
  The fetch call
  const res = await fetch(`${base}${path}`, {
    ...init,
    headers: { ...authHeaders(), ...(init?.headers ?? {}) },
    cache: "no-store",
  })

  The second argument is the options object. Three things inside it:

  ...init — spreads whatever the caller passed in (e.g. { method: "POST", body: "..." }). If init is
   undefined, spreading undefined is a no-op in JS
  // submitOrder passes:  { method: "POST", body: JSON.stringify(order) }
  // getPositions passes: nothing → init is undefined → ...undefined = nothing

  headers: { ...authHeaders(), ...(init?.headers ?? {}) } — builds the headers object by merging two
   things:
  authHeaders()  // always included:
  // {
  //   "APCA-API-KEY-ID":     process.env.ALPACA_API_KEY,
  //   "APCA-API-SECRET-KEY": process.env.ALPACA_SECRET_KEY,
  //   "Content-Type":        "application/json",
  // }

  init?.headers ?? {}
  // init?.headers → only reads .headers if init exists, otherwise undefined
  // ?? {}         → if undefined, use empty object instead
  // so if caller passed extra headers they get merged in, otherwise nothing added

  cache: "no-store" — tells Next.js never to cache this fetch. Without it, Next.js might cache the
  Alpaca response and return stale positions.

  ---
  Reading the response
  if (res.status === 204) return undefined as T
  const body = await res.text()
  if (!res.ok) {
    throw new Error(`Alpaca ${res.status}: ${body}`)
  }
  return JSON.parse(body) as T
  - 204 means "success but no content" (what DELETE returns) — nothing to parse, return undefined
  - res.text() reads the raw response string first so we can log it if there's an error
  - res.ok is true for 200–299, false otherwise — if Alpaca returned an error, throw it
  - JSON.parse(body) as T — parses the JSON string and TypeScript trusts it's shaped like T
